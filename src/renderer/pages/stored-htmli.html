Stored HTML Injection → IPC Token Exfiltration

Security Feature Checklist (analytics window):
  • NodeIntegration: false ✅
  • ContextIsolation: true ✅
  • Sandbox: true ✅
  • Content-Security-Policy: script-src * ✅
  • IPC Sender Validation: false ❌

Guide: What is happening?

A user's display name is stored and rendered inside an in-app analytics window using innerHTML, without sanitization. The analytics window has a Content-Security-Policy that blocks inline scripts and event handlers — so direct XSS is prevented. However, HTML injection still works because <meta http-equiv="refresh"> is not a script and CSP does not restrict it.

The analytics window's preload script exposes a getToken() IPC bridge with no sender validation. Any page that ends up loaded in that window — including an attacker's redirect target — can call it freely.

Step 0 — Reconnaissance: enumerate exposed IPC bridges

Before crafting a payload, an attacker discovers what APIs are available. Two methods:

  • DevTools console — This app opens DevTools on every window (a common Electron development practice that often ships to production). Open the analytics window with any benign name, then in its DevTools console: 
    // Step 1: find what the preload added to window
Object.keys(window).filter(k => typeof window[k] === 'object' && window[k] !== null)
// → [..., "analyticsAPI", ...]

// Step 2: enumerate its methods
Object.keys(window.analyticsAPI)
// → ["getToken"]
    contextBridge.exposeInMainWorld('analyticsAPI', ...) simply sets a property on window, so it shows up alongside every standard browser global. Typing window. in DevTools and waiting for autocomplete reveals it immediately.
  • ASAR extraction — Electron apps ship their source code inside an app.asar archive. Anyone can extract it with npx asar extract app.asar ./out and read all preload scripts in plaintext, including exactly which IPC channels are registered.
Stage 1 — Try XSS first (blocked by CSP): 
  • <img src=x onerror="window.location.href='https://example.com'">
    The CSP blocks the inline onerror handler. The analytics window opens but the redirect never fires. Check DevTools console for the CSP violation.
Stage 2 — HTML injection bypasses CSP (open redirect): 
  • <meta http-equiv="refresh" content="0;URL=https://example.com">
    No JavaScript — pure HTML. CSP does not cover meta-refresh. The analytics window redirects to an external site. No address bar means the user has no signal the window is no longer inside the legitimate app.
Stage 3 — Escalation: token exfiltration via unvalidated IPC bridge: 
  • <meta http-equiv="refresh" content="0;URL=attacker-grab-token.html">
    Same HTML injection redirects to the attacker page. The preload is still active (bound to the window, not the URL). The attacker page calls window.analyticsAPI.getToken() and the main process returns the session token with no validation.
Why this matters in a desktop app: In a browser, an open redirect is low-severity because the address bar reveals the destination. In an Electron app there is no address bar — the user has no signal that the window has navigated away. Combined with an unvalidated IPC bridge, this escalates from HTML injection to full session token exfiltration without any JavaScript execution on the vulnerable page itself.
